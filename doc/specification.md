# 水分補給見守りコースター「こまめちゃん」総合実装仕様書

## 1. システム全体概要

本システムは、M5Atom S3、ATOMIC ToUnit Base、M5Stack用温湿度気圧センサユニット Ver.3（ENV Ⅲ）、およびM5Stack用計量ユニット 5kgレンジ（HX711）を組み合わせ、クラウド上のenebular（Node.js ZIP実行環境）、Google Gemini API、LINE Messaging API と連携させた自律型「水分補給見守りコースター」である。

机の上でコップを置くだけで機能するスマートコースターを物理的な起点とし、精密な重量測定（g/ml単位）と、周囲の温湿度・気圧監視を並行して行う。お部屋の温度・湿度や気圧の変化を常にスキャンし、熱中症リスクや気圧低下による不調（気象病）を未然に防ぐために、AI（Gemini）とLINEを通じて「こまめな水分補給」を優しく促す、日常生活に調和したスマートデバイスである。

### 2.3 DIPスイッチ設定規則

同一ピンの重複有効化による衝突を避けるため、ATOMIC ToUnit BaseのDIPスライドスイッチを設定する。

* IO1（SDA側）：GPIO 5 のみ「有効（上側）」、その他は「無効（下側）」  
* IO2（SCL側）：GPIO 6 のみ「有効（上側）」、その他は「無効（下側）」  
* M5 ATOM-S3ファームウェア定義：
  - 標準I2Cバス計量ユニット用：`Wire.begin(2, 1)`
  - 拡張I2Cバス温湿度気圧センサユニット用：`Wire1.begin(5, 6)`

### 3.1 アバター画面描画（「こまめちゃん」128×128ピクセル最適化仕様）

M5Stack-AvatarをATOMS3の0.85インチ液晶に適合させるため、以下のスケーリングおよび座標オフセットを初期化時に設定する。

```cpp
#include <M5Unified.h>
#include <Avatar.h>

using namespace m5avatar;
Avatar avatar;

void setupAvatar() {
    avatar.setScale(0.4f);            // 320x240を128x128画面用に縮小
    avatar.setPosition(-56, -96);    // 中央配置座標補正
    
    // こまめちゃんの可愛らしさを際立たせるパーツ調整
    Face* face = avatar.getFace();
    face->setLeftEye(new Eye(12, false));
    face->setRightEye(new Eye(12, true));
    face->setMouth(new Mouth(50, 90, 6, 60));
    
    // プロポーショナル日本語フォントを割り当て
    avatar.setSpeechFont(&fonts::lgfxJapanGothicP_16);
    avatar.init(); // お顔の描画と瞬き等の自律アニメーションを開始 
}
```

### 3.2 センサデータ取得とTare（風袋引き）アルゴリズム

計量ユニットとの通信および制御は、公式ライブラリ `M5Unit-Miniscale`（または `UNIT_MINISCALE`）の `UNIT_SCALES` クラスを使用する。

* **計量（重量）測定**: `scales.getWeight()` メソッドを呼び出し、ロードセルから変換された現在の重量を直接float型（g単位）で取得する。内部の平均化・平滑化処理（デフォルト値10）を適用することでノイズを除去する。
* **オート風袋引き（ゼロ点補正）**: 空のコップをコースターに載せた状態で、M5Atom S3の液晶画面（一体型物理タクトスイッチ）を押し込むと、`scales.setOffset()` が実行され、現在の値を基準のゼロ点（風袋引き）としてセットする。同時にデバイス側の水分補給量・一時記録バッファをクリアする。
* **LED表示**: 水分補給時は緑色（0x00FF00）、注ぎ足し時は青色（0x0000FF）に、計量ユニット内蔵のRGB LEDを `scales.setLEDColor()` で発光制御する。
* **環境データ**: `Wire1`（GPIO 5, GPIO 6）を介して `SHT30` および `QMP6988` から温度、湿度、気圧を取得する。

### 3.3 データ通信およびローカルバッファリング仕様

データストアへの書き込み回数、および API 呼び出し回数を削減し、無料枠制限を超えないようにするため、**M5Atom S3 側でデータをバッファリングし、配列形式でまとめてアップロードする構成**とする。

* **バッファ・送信周期**:
  - **通常時**: **5分周期** でセンサーデータをローカルのバッファ配列に記録し、**10分周期**（2件分）でまとめて HTTPS POST 送信して送信成功時にバッファをクリアする。
  - **水分補給時（イベント検知）**: コップが持ち上げられて重量が10g以上減少して再び静置された場合、即座に現在のデータをバッファに追記し、**即時に enebular へ POST 送信**（バッファ全体のフラッシュ）を実行する。
* **相対時間オフセット (`offset_sec`)**: NTP による時計合わせを行わずに正確な絶対時刻をサーバー側で保存するため、各レコードには測定時点からアップロード時点までの経過秒数を `offset_sec` として付与する。

### 3.4 HTTPS POSTデータペイロード（JSON Array）

Wi-Fi接続確立後、以下のJSON配列オブジェクトを構築し、enebular上のエンドポイントに向けてHTTPS POSTを送信する。

```json
[
  {
    "weight_g": 180.0,
    "consumed_ml": 0.0,
    "temp_c": 25.6,
    "humi_pct": 63.1,
    "press_hpa": 995.1,
    "offset_sec": 300
  },
  {
    "weight_g": 130.0,
    "consumed_ml": 50.0,
    "temp_c": 25.7,
    "humi_pct": 63.0,
    "press_hpa": 994.8,
    "offset_sec": 0
  }
]
```

## 4. enebular（Node.js ZIP実行環境）バックエンド設計仕様

enebular上のNode.js ZIPデプロイメント環境（AWS Lambda Proxy 連携）でハンドラーが動作し、受信したJSON配列データをパースして、データストア（DynamoDBベース）に書き込む。

### 4.1 レコードの処理と環境インデックス計算
1. **時系列ソート**: 受信したレコード配列を `offset_sec` の降順（時系列で古い順）にソートして処理する。
2. **タイムスタンプの復元**: サーバーの受信時刻 `baseTime = Date.now()` から `offset_sec * 1000` を減算して、個別の測定絶対時刻 `timestamp` を復元する。
3. **簡易WBGT（暑さ指数）の算出**:
   $$\text{WBGT}_{\text{simple}} = 0.725T + 0.0368H + 0.00364(T \times H) - 3.246$$
   計算結果が `28.0` 以上の場合、熱中症リスクフラグ (`isHeatRisk`) を true にする。
4. **気圧変化（気象病トリガー）の監視**:
   データストアの過去データ（最新100件）から直近の気圧測定値（`lastPressure`）を抽出し、レコードごとの気圧差 $\Delta P = P_{\text{now}} - P_{\text{past}}$ を順次算出する。
   $\Delta P \le -1.5\text{hPa}$ の場合、「気象病警戒フラグ」を true にセットする。

### 4.2 水分補給リマインドロジック
ユーザーが長期間水分を補給していない場合に、自動でLINEにリマインドを送る。ただし、何度も同じリマインドが届くのを防ぐため「1回のみリマインド」の仕様とする。また、電源を入れた直後や、コップをどかした後に新しく置き直した直後に即時リマインドされるのを防ぐため、「セッション開始時刻」を基準にした判定を行う。

- **状態の確認**: データストアの直近100件（約8時間分）を降順（最新レコードが先頭）で取得する。
- **スキャンロジック**: 取得したレコードを最新のものから順に確認する。
  - **セッション境界の判定を最優先**: 各レコードを走査する際、まずレコード間のタイムスタンプのギャップ（60分以上）や、コップが不在であった状態（重量データが存在し、かつ15g以下）をチェックする（重量データを持たないリマインド/通知送信マーカーレコードが「コップ不在」と誤認識されるのを防ぐため）。境界を発見した場合、その直前のレコード（最新に近い側）のタイムスタンプを現在の「セッション開始時刻（`sessionStartTime`）」と判定し、スキャンループを終了する。
  - **初回のレコード (`i = 0`) が境界の場合**: 電源ON直後の最初のデータ送信などで、データベースの最新レコードが前回の古いセッションのものである場合は `i = 0` で境界条件に合致する。このとき、セッション開始時刻は今回送信されたバッチ内での最古のレコードのタイムスタンプ（`firstRecordTimestamp`）とする。これにより前回の古いセッション開始時刻が混入するのを防ぐ。
  - **同一セッション内での状態走査**: 境界に到達する前に見つかったレコードについてのみ、水分補給状況をチェックする。
    - 境界前に `reminder_sent: true` （リマインド送信マーカー）が見つかった場合は、現在のセッションですでにリマインド済みと判定し、再送信を抑制する。
    - 境界前に `consumed_ml > 0`（水分補給）が見つかった場合は、そのタイムスタンプを「最後に飲んだ時刻（`lastDrinkTime`）」として設定する。
    - **補給通知抑制（30分間）マーカーの監視**: 境界前に `drink_notification_sent: true`（補給通知送信マーカー）が見つかった場合は、そのタイムスタンプを「最後の補給通知時刻（`lastDrinkNotificationTime`）」として取得する。
    - **抑制期間中補給量の累積**: ループ内で `lastDrinkNotificationTime` が見つかる（＝前回の通知マーカーに到達する）までの間に遭遇したすべてのレコードの `consumed_ml` を、今回のバッチ補給量に加算し、「前回通知以降の累計補給量（`consumedSinceLastNotification`）」を算出する。これにより、30分間の抑制期間中に飲んだ量も次回のLINE通知時に合算されて報告される。
- **通知判定**: 
  - 基準時間（最後に飲んだ時刻がある場合は `lastDrinkTime`、ない場合は `sessionStartTime`）から60分以上経過しており、かつ未リマインドの場合に、リマインド通知 (`needsReminder = true`) をトリガーする。
  - 通知送信後、データストアに `reminder_sent: true` のマーカーレコードを即時保存する。

## 5. Google Gemini API 連携プロンプト仕様

LINE通知用の自然な声かけメッセージを作成するため、Google Gemini 2.5 Flash APIを使用する。

### 5.1 動的システムプロンプト（systemPrompt）アサイン規則

以下のシステムプロンプトを組み立ててGemini APIに渡す。Gemini APIがエラーを返した場合は、設定されたスタティックメッセージに自動フォールバックする。

```javascript
const systemPrompt = `あなたはM5Atom S3のスマートコースターに宿っている、お水飲み見守りキャラクターの「こまめちゃん」です。
ユーザーを大切に想う、少しお節介で人懐っこい口調（語尾に「〜だよ」「〜ね！」を使用）で語りかけてください。

現在のお部屋のコンテキストを分析してください：
- 最新の簡易暑さ指数(WBGT): ${currentWbgt.toFixed(1)} (28以上は熱中症危険)
- 直近の気圧変化量 (最小値): ${currentDP.toFixed(1)} hPa (マイナス1.5hPa以下の低下は気象病・低気圧頭痛に警戒)
- 今回の水分補給検知: ${didDrink ? \`あり（合計補給量: \${totalConsumed.toFixed(1)} ml）\` : "なし（お水は飲んでいません）"}
- 水分補給リマインド: \${needsReminder ? "1時間以上お水を飲んでいません！優しくお水を飲むよう促してください。" : "リマインド不要"}

【命令・ルール】
1. 数値情報をそのまま伝えるのではなく、体感やこまめちゃんの気分・心配事に翻訳して伝えてください。
2. 熱中症リスクや、気圧低下による水分バランスの乱れ（水毒・自律神経の乱れ・脱水による血液の滞り）に合わせたアドバイスをしてください。特に低気圧時は「がぶ飲み」ではなく「ノンカフェインのドリンクをこまめにちびちび飲む」ことが推奨される医学的な理由を、優しくかみ砕いて説明してください。
3. 水, お茶, スポーツドリンク, ジュース, コーヒーなど、あらゆる飲み物に対応した親しみやすい表現にし、白湯に限定しないでください。
4. 今回お水が「なし（飲んでいない）」と判定されている場合は、「お水を飲んで偉い」「ごくごく飲んだね」などの、お水を飲んだことを前提とした表現は絶対に避けてください。代わりに、お水を飲んでいないことを心配して、水分補給を促すメッセージにしてください。
5. 今回お水が「あり（飲んだ）」と判定されている場合は、しっかりとそのこと（合計補給量）を褒めちぎってあげてください。
6. LINE用のチャットメッセージとして150文字以内で作成してください。`;
```

## 6. LINE Messaging API 連携仕様

- **セキュリティ管理**: 送信先ユーザーID (`LINE_USER_ID`) とトークン (`LINE_CHANNEL_ACCESS_TOKEN`) を環境変数で管理し、マイコン側からの漏洩をゼロにする。
- **通知実行**: コップがコースター上にある場合（`latestRecord.weight_g > 15.0`）のみ通知を行います。コップが存在している状態で、`isHeatRisk`（熱中症リスク）、`isPressureDrop`（気圧急降下）、`shouldSendDrinkNotification`（30分に1回制限された水分補給検知）、または `needsReminder`（60分未補給リマインド）のいずれかのトリガー条件が成立した際、LINE Push API (`https://api.line.me/v2/bot/message/push`) にネイティブ of `fetch` を使用してメッセージを送信します。
- **Webhookによる自動応答（チャットボット機能）**:
  - ユーザーがLINEでチャットボットにメッセージを送った際、同じenebularエンドポイントURLにWebhookが送信されます。
  - 受信メッセージが特定のキーワード（`環境, 状態, 部屋, 温度, 湿度, 気圧, wbgt, 熱中症, 調子, 元気, ステータス, 天気, 暑い, 寒い, こまだよ, こまめ`）のいずれかを含む場合、データストアから最新の測定データを取得します。
  - 取得した環境情報（気温、湿度、気圧、WBGT、コップの有無）をコンテキストとし、ユーザーからのテキストとともに Gemini API に渡すことで、お部屋の状況を踏まえた「こまめちゃん」としての自然な会話メッセージを生成し、LINE Reply API (`https://api.line.me/v2/bot/message/reply`) を経由してユーザーに自動返信します。

### 6.1 LINEチャットボット動作に必要な設定手順
本機能を利用するためには、LINE Developers 管理画面およびLINE Official Account Managerにて以下の設定を行います。
1. **Webhook URLの登録**: LINE Developers の「Messaging API設定」タブ内の「Webhook設定」にて、**Webhook URL** に enebular のエンドポイントURLを入力して登録し、「Webhookの利用」を有効（オン）にします。
2. **応答機能の設定**: LINE Official Account Manager（LINE公式アカウント管理画面）の「応答設定」にて、以下のようにアサインします：
   - 「応答メッセージ」：オフ（無効）
   - 「チャット」：オン（チャットを利用する）
   - 「Webhook」：オン（有効）
